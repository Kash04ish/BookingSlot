# Delivery Slot Booking System

A small full-stack app to view delivery slot availability across the next seven days, book a slot, prevent overbooking under concurrent requests, and suggest the next available slot at a click.

Built as a take-home case study for an Associate Software Engineer role.

## Tech stack

| Layer            | Choice                                    |
| ---------------- | ----------------------------------------- |
| Framework        | Next.js 14 (App Router, TypeScript)       |
| UI               | Tailwind CSS, sonner toasts, lucide-react |
| Forms/validation | react-hook-form + Zod                     |
| Database         | SQLite via Prisma ORM 5                   |
| Tests            | Vitest                                    |
| Tooling          | ESLint, Prettier defaults from Next.js    |

## Getting started

Requirements: Node 18+ and npm.

```bash
git clone <repo-url>
cd booking-slot
npm install
cp .env.example .env
npx prisma migrate dev --name init   # creates the SQLite DB + applies migrations
npm run seed                          # seeds 7 days x 9-18h hourly slots, capacity 5
npm run dev                           # http://localhost:3000
```

Run tests:

```bash
npm test
```

## Features

- **Stores delivery slots with capacity** — modeled in Prisma as `Slot { startsAt, endsAt, capacity }`. Seeded with 63 slots covering the next 7 days, hourly between 09:00 and 18:00, capacity 5 each.
- **Books slots** — `POST /api/bookings` creates a `Booking` linked to a `Slot`. Input is validated with the same Zod schema on client and server.
- **Prevents overbooking** — `bookSlot` runs inside a serialized Prisma transaction so concurrent requests cannot both pass the capacity check. A Vitest test fires 12 concurrent requests at a capacity-5 slot and asserts that exactly 5 succeed.
- **Suggests the next available slot** — `GET /api/slots/next-available` returns the earliest future slot with room. The UI scrolls it into view and highlights it for a few seconds.
- **Displays availability** — the homepage groups slots by day and shows `booked/capacity` with a green / amber / red status pill. Full slots are disabled with a tooltip.

## Architecture

UI components fetch from Next.js Route Handlers under `/api/*`. Route handlers delegate all domain logic to plain functions in `src/lib/` (`bookSlot`, `listSlots`, `getNextAvailable`), which talk to SQLite through Prisma. The same Zod schemas in `src/lib/schemas.ts` are used by both the form (`react-hook-form` resolver) and the API route, so a malformed input is rejected consistently in either place. Errors flow through a `BookingError` type with a `code` field that the route layer maps to HTTP status codes.

## Overbooking strategy

`bookSlot` enforces the capacity invariant with two layers:

1. **Per-slot in-process mutex** — concurrent requests for the same `slotId` are serialized in JavaScript before they reach the database, so the capacity check and the insert always happen atomically together. This is the layer the concurrency test exercises directly, and it also keeps SQLite from surfacing transient lock-contention errors under bursty load.
2. **Prisma transaction** — inside the lock, the read-and-insert run inside `prisma.$transaction`, so even if the process is restarted mid-operation, the booking is either fully written or not written at all.

The `'prevents overbooking under concurrent requests'` test in `src/tests/booking.test.ts` confirms this by firing twelve concurrent `bookSlot` calls at a capacity-5 slot and asserting that exactly five succeed and the other seven reject with `BookingError({ code: 'SLOT_FULL' })`.

Trade-off: the in-process mutex assumes a single Node process. For a horizontally-scaled deployment, the right next step is to move the lock into the database via Postgres row-level locking (`SELECT ... FOR UPDATE`) or an advisory lock. For a single-tenant demo this is the right balance of simplicity and provable correctness.

## Validation and error handling

- Zod schemas live in `src/lib/schemas.ts` and are reused on both client (via `react-hook-form` resolver) and server (re-validated inside `bookSlot` as defense in depth).
- Consistent error contract: `{ error: { code, message, details? } }`. Codes:
  - `INVALID_INPUT` → HTTP 400
  - `SLOT_NOT_FOUND` → HTTP 404
  - `SLOT_FULL` → HTTP 409
  - `INTERNAL` → HTTP 500
- The UI handles each code with a specific user-facing toast — for example, on `SLOT_FULL` it suggests trying the next available slot and closes the modal.

## Assumptions

- Slots are seeded for the next 7 days, hourly between 09:00 and 18:00, capacity 5. The seed script is idempotent (`upsert` by `startsAt`).
- No authentication; this is a single-tenant demo.
- "Next available slot" = the earliest slot with `startsAt > now()` and `bookedCount < capacity`.
- Times stored as UTC; rendered in the browser's local timezone via `date-fns`.
- A booking captures name, email, optional phone, optional notes.
- One booking equals one unit of capacity (no party-size field).

## AI-assisted development

I used **Claude Code (Claude Opus 4.7, 1M context)** as my primary AI pair for this build. It helped me:

- Settle on the stack (Next.js + Prisma + SQLite) under the constraint that a reviewer should be able to run the project in under a minute.
- Reason through the overbooking problem and pick a transaction isolation level that works on SQLite, rather than reaching for a Postgres-only `SELECT ... FOR UPDATE`.
- Generate the concurrency test that proves the overbooking guarantee, and write the seed script + Vitest setup so that tests run against an isolated `test.db`.
- Draft this README and keep the error contract consistent across API routes.

What I had to push back on: an early suggestion was a naive check-then-insert without a transaction, which would be racy under concurrent load. I asked for a revision specifically for the concurrent case, then verified the fix with a deliberate `Promise.allSettled` of 12 requests in `booking.test.ts`. I also tightened the UX (showing "Filling fast" at ≤2 spots left, scrolling to a suggested slot, sonner toasts that recommend the next available slot on a 409) — none of which the AI proposed unprompted.

The main challenge was scoping. It is easy to keep adding features (cancellation, admin view, calendar export, real auth). I kept the surface area small and the booking logic provably correct, since the rubric weights correctness, validation and code quality heavily.

## What I would do next

- Real auth so "My bookings" becomes meaningful, plus per-user rate limiting on `POST /api/bookings`.
- Postgres in production with row-level `SELECT ... FOR UPDATE` for finer-grained locking instead of serialized transactions.
- An admin UI to create/cancel slots and adjust capacity.
- E2E coverage with Playwright (book, fill to capacity, suggest next, cancel).
- ICS calendar export on booking confirmation.
